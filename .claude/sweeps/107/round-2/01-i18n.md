## Domain 5 — Backend tail (orbit-api English display sentences)

Audited `eee06ae` on `chore/107-code-health-sweep`. ERRORS-API landed the Application-layer half: ~148-150 `Result.Failure` sites now carry stable codes via `ErrorMessages.*`/`AppError` (Code+Message), and `ResultActionResultExtensions.ToErrorResult()` correctly emits `{ error, errorCode }`. BUT controllers still hand-roll `new { error = result.Error }` at 59 sites (grep-counted), dropping `errorCode` before it reaches the wire. Spot-verified by direct read. Severities reflect what users actually hit.

- **HIGH · orbit-api src/Orbit.Api/Controllers/AuthController.cs:48,97,158,224,277,326,349 (7 sites) · issue #107 agent-1 + rule 1** — auth endpoints serialize `new { error = result.Error }` and DROP `result.ErrorCode`, even where the command sets real codes (`VerifyCodeCommand` emits `TOO_MANY_ATTEMPTS`/`CODE_EXPIRED`/`INVALID_VERIFICATION_CODE`). Confirmed `Result.ErrorCode` exists and `ToErrorResult()` would emit it. Fix: return `result.ToErrorResult(statusCode)` (or add `errorCode = result.ErrorCode` to the payload). Highest-traffic localized-error path (login/verify/deletion).
- **HIGH · orbit-api src/Orbit.Api/Controllers/** — 44 more `new { error = result.Error }` bypass sites (59 total − 15 OAuth; AuthController's 7 counted above → ~37 across the rest) · issue #107 agent-1 + rule 1** — by file: ChatController 10, AiController 7, TagsController 3, ReferralController 3, SyncController 2, SubscriptionController 2, NotificationController 2, HabitsController 2, ChecklistTemplatesController 2, SupportController 1, ProfileController 1, CalendarController 1, ApiKeysController 1. Each drops an available `errorCode`. These cover habit/goal/tag mutations and subscription/notification flows users hit routinely → pt-BR users see English. Fix: route every failure through `ToErrorResult()` instead of hand-rolling the payload. (HIGH because the client mapping for these codes ALREADY EXISTS — see client note — so this single backend change unlocks localization for the whole catalog.)
- **MED · orbit-api src/Orbit.Api/Controllers inline bare-English payloads · issue #107 agent-1** — strings with NO code at all: ChatController.cs:196,205 "Chat history too large", :202,216,228 "Invalid chat history format", :241,248 "Invalid client context format" (codes EXIST as `ErrorMessages.ChatHistoryTooLarge`/`InvalidChatHistory`/`InvalidClientContext` but the controller uses literals); ChatController.cs:40,85 `"Message must be between 1 and {N} characters"` (use `ErrorMessages.MessageTooLong`); AiController.cs:181,284 "Pending operation not found or expired." (use `ErrorMessages.PendingOperationNotFound`); AiController.cs:323 `firstError.ErrorMessage` (raw FluentValidation prose, no code); CalendarController.cs:75 "Enabled is required."; SyncController.cs:329 "No mutations provided.", :332 "Maximum 100 mutations per batch." Fix: replace literals with the catalog `AppError` and emit via `ToErrorResult()`.
- **MED · orbit-api src/Orbit.Api/Controllers/AiController.cs:335,346,362,377 · issue #107 agent-1 + rule 1** — these DO reference the catalog (`ErrorMessages.ClarificationNotFound` etc.) but wrap it as `new { error = ErrorMessages.X }`. Since `ErrorMessages.X` is an `AppError` record, this serializes `error` as an OBJECT `{ code, message }` with NO top-level `errorCode` — the client's `extractBackendErrorCode` reads `data.errorCode`, so the code is unreachable AND the `error` field is malformed. Fix: emit `Result.Failure(ErrorMessages.X).ToErrorResult(status)` so the shape is `{ error: string, errorCode: string }`.
- **MED · orbit-api src/Orbit.Application/** validators (~64 `.WithMessage("English prose")` sites) · issue #107 agent-1** — FluentValidation messages flow to clients as raw field errors; the client substring-matches English (`CONTEXTUAL_RULES`) and unmatched prose renders verbatim. Some validators already use `ErrorMessages.*` (e.g. ResolveClarificationRequestValidator) — those are correct; the residual bare-string validators (CreateApiKeyValidator, ProcessUserChatCommandValidator, CreateChecklistTemplateCommandValidator, BulkCreateHabitsCommandValidator, etc.) still emit prose without a code/key. Fix: stable validation codes the client maps.
- **LOW · orbit-api src/Orbit.Api/Controllers/OAuthController.cs:189,197,202 (+117,120) · issue #107 agent-1** — `"Invalid or expired Google sign-in token"`, `"Could not retrieve email from Google account"`, `"Google token was not issued for this application"` returned as `new { error = "..." }` on the OAuth authorize flow. Round-1 classed OAuthController as OAuth-wire-format-exempt; that is correct ONLY for the spec-coded responses (lines 246 `unsupported_grant_type`, 253 `invalid_grant`+`error_description`, 269 `server_error`). These 5 are English sentences a user could see if surfaced by a consent client, not OAuth machine codes — borderline. LOW: this endpoint is largely redirect/machine-driven and primary login uses `GoogleAuthCommand` via AuthController. Note, don't prioritize over the controller-bypass fix.
