# Sweep 9 ‚Äî Test coverage gaps + test quality (issue #107, round 1)

Read-only audit of both working trees (orbit-ui-mobile @ main clean, orbit-api). Method: full inventory of web/mobile hooks, shared utils/validation, web+mobile lib/stores, all 113 Application handlers, 48 validators, 9 Application services, 34 Domain entity files ‚Äî diffed against every `*.test.ts(x)` file and `tests/Orbit.*.Tests` via symbol search. Per instruction, the absence of integration/E2E suites is NOT reported (deliberately deleted; `tests/Orbit.IntegrationTests/` is an empty shell with no tracked files).

## Findings

- HIGH ¬∑ `apps/mobile/lib/google-auth.ts` (exchangeGoogleSession, mergeRequestIdIntoPayload, browser sign-in flow; 188 lines) ¬∑ Mobile Google OAuth login path has ZERO test coverage: no test references `signInWithGoogle`, `exchangeGoogleSession`, or `mergeRequestIdIntoPayload` anywhere (searched all `apps/mobile/__tests__`; `google-auth-callback.test.ts` covers only the sibling callback module, and the earlier `lib/google-auth` grep hit was a substring match on `lib/google-auth-callback`). This is the only Google login path on Android. ¬∑ Suggested: unit-test `mergeRequestIdIntoPayload` (null/non-object/existing-requestId payloads) and `exchangeGoogleSession` success + non-OK error shaping with stubbed `fetch` (its own boundary), plus the WebBrowser result-type branches with mocked `expo-web-browser`.

- HIGH ¬∑ `apps/web/app/api/auth/{google,send-code,verify-code,session,logout}/route.ts` + `apps/web/lib/auth-proxy.ts` (BFF auth routes + helpers) ¬∑ The login cookie boundary has zero tests: no file in `apps/web/__tests__` references any auth route or imports `lib/auth-proxy` (only the generic catch-all is covered by `__tests__/app/api/proxy-route.test.ts`; `lib/auth-api.test.ts` covers the cookie helpers, not the routes). Untested behavior includes the documented contract "sets httpOnly auth + refresh cookies, then returns user data without the token" (token-stripping), error payload shaping via `buildAuthErrorPayload`, and request-id propagation. `auth-proxy.ts` is five pure, trivially testable functions. ¬∑ Suggested: route tests mirroring `subscriptions-checkout-route.test.ts` (mock fetch, assert cookies set, token absent from body, status/error passthrough) + a small `auth-proxy.test.ts`.

- MED ¬∑ `orbit-api src/Orbit.Application/Goals/Services/GoalStreakSyncService.cs` (NeedsPassiveSync, CalculateCurrentStreak, SyncCurrentStreak[IfNeeded]) ¬∑ Streak math with zero direct unit tests: `git grep GoalStreakSyncService tests/` ‚Üí 0 hits. Coverage is only indirect via `GetGoalsQueryHandlerTests`/`GetGoalMetricsQueryHandlerTests` (they touch `SyncStreakProgress`/`StreakSyncedAtUtc`); the date-boundary logic (`syncedDate < userToday`, null synced date), Min()-across-linked-habits, and empty-habit-list null path are not asserted in isolation, and `LogHabitCommand`/`SkipHabitCommand` call sites depend on them too. ¬∑ Suggested: `tests/Orbit.Application.Tests/Services/GoalStreakSyncServiceTests.cs` covering same-day no-resync, stale-sync resync, non-streak/non-active goals, min-of-habits, zero linked habits.

- MED ¬∑ `orbit-api src/Orbit.Application/Profile/Queries/ExportUserDataQuery.cs` (ExportUserDataQueryHandler) ¬∑ The ONLY one of 113 Application handlers with zero test references (`git grep ExportUserData tests/` ‚Üí empty; every other defined handler name appears in `tests/`). Real logic: 7-repository aggregation, logs grouped/ordered per habit and goal, user-not-found failure path, plan mapping. Account data export is a privacy/compliance surface. ¬∑ Suggested: `Queries/Profile/ExportUserDataQueryHandlerTests.cs` ‚Äî user not found, empty account, habits-with-logs ordering, pro/free plan mapping.

- MED ¬∑ `orbit-api tests/Orbit.Infrastructure.Tests/Services/HabitDueDateAdvancementServiceTests.cs:301-351` (QueryFilter_HabitDueDateAtCutoff_NotIncluded, QueryFilter_HabitDueDateBeforeCutoff_Included, TimezoneGuard_* x3) ¬∑ Tests that re-implement the logic under test: each computes `var shouldAdvance = habit.DueDate < cutoff;` / `var shouldSkip = habit.DueDate >= userToday;` inline and asserts on its own expression ‚Äî the service is never invoked, so these 5 tests pass regardless of what the advancement job actually does. The overdue-advancement cutoff/timezone guard (sensitive: advancement must only touch BAD habits / not hide overdue) is effectively uncovered by them. ¬∑ Suggested: drive the service's real filter/guard (extract the predicate into a testable method on the service and assert through it), or delete the tautologies.

- MED ¬∑ `apps/mobile/app/(tabs)/profile.tsx` (account-deletion flow: deleteStep 'confirm'‚Üí'code'‚Üí'deactivated', 6-digit code entry, openDeleteModal/backToDeleteConfirmStep, loading/error state) ¬∑ Zero mobile tests for the delete-account state machine: `grep -i delete apps/mobile/__tests__/screens/profile-screen.test.tsx` ‚Üí no hits; only `intentional-offline-ux.test.tsx` touches the symbol (offline gating). - MED ¬∑ `apps/mobile/hooks/use-login-code-entry.ts` (useLoginCodeEntry) ¬∑ Zero mobile tests (no symbol hit in `apps/mobile/__tests__`); web twin has a dedicated `apps/web/__tests__/hooks/use-login-code-entry.test.ts`. - MED ¬∑ `apps/mobile/hooks/use-chat-reward.ts` (useChatReward) ¬∑ Mobile-only rewarded-ad ‚Üí `POST API.subscription.adReward` ‚Üí patches cached `Profile.aiMessagesLimit`; gating via `canWatchRewardAd` (ads enabled + AdMob ready + daily cap + not loading). - MED ¬∑ `apps/mobile/hooks/use-review-reminder.ts` (useReviewReminder.shouldShow + requestReview) ¬∑ Zero test references for the hook (only `stores/review-reminder-store.test.ts` covers the store). - MED ¬∑ `apps/web/hooks/use-dismiss-guard.ts` + `apps/mobile/hooks/use-dismiss-guard.ts` (useDismissGuard) ¬∑ The dirty-form discard guard (requestDismiss/confirmDismiss/cancelDismiss + showDiscardDialog) protecting modals from data loss has zero tests on BOTH platforms ‚Äî the symbol appears in no test file in the repo. ¬∑ Suggested: one renderHook spec per platform: clean ‚Üí dismisses immediately; dirty ‚Üí opens dialog; confirm ‚Üí dismisses and closes; cancel ‚Üí keeps editing.

## Not findings (checked, clean)

- All 48 Application validators and 112/113 handlers are referenced in `tests/Orbit.Application.Tests` (symbol diff of `class \w+Handler|Validator` vs test refs).
- Domain entities with behavior (Habit, Goal, User, Tag, Referral, StreakFreeze, ApiKey, Notification, ChecklistTemplate, agent entities, ValueObjects ChecklistItem/ScheduledReminderTime) all have domain-test coverage; the uncovered entities (SentReminder, SentSlipAlert, GoalProgressLog, ProcessedPlayNotification, AppConfig, HabitLog...) are anemic Create()-only rows ‚Äî no logic to test.
- No skipped/focused tests anywhere: zero `it.skip/.only/xit/xdescribe` in the frontend, zero `[Fact(Skip=...)]` in orbit-api (the `Skip =` grep hits are local variables).
- No `expect(true).toBe(true)`-style assertions; no snapshot tests at all.
- Mobile tests follow the "mock at hook level" rule; the `global.fetch` stubs in `api-client.test.ts`, `version-check.test.ts`, `auth-store.test.ts` mock the unit's own boundary (those modules call fetch directly), which is correct.
- Shared `utils/` is fully covered except pure constants/lookup maps (`ai-summary.ts` = 1 const, `achievement-emoji.ts` = emoji map) ‚Äî trivial, not findings.
- `useBulkActions` (initially suspected) is exercised end-to-end through `today-page.test.tsx` / `today-screen.test.tsx` with only leaf mutations mocked (descendant-suppression behavior asserted) ‚Äî adequate.
