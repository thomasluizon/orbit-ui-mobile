## Findings

1. **HIGH** · web `apps/web/components/habits/create-habit-modal.tsx:360` + mobile `apps/mobile/components/habits/create-habit-modal.tsx:313` (and `apps/mobile/components/habits/habit-form-fields.tsx:140`) · dim 1+5 (submit gating) · Web gates submit on full `formState.isValid` (live zodResolver) while mobile gates on buffered title presence only. With an emptied/zero frequency quantity, web silently disables the button with no rendered error; mobile keeps it enabled and toasts `habits.form.frequencyRequired`. Fix: pick ONE gating contract for both — gate on title presence only (`title.trim().length > 0`) and let the shared `validateAll` toast carry everything else (mobile model), or gate both on schema validity with a visible inline reason. Encode it once (e.g. a shared `isHabitFormSubmittable(values)`), import on both.
2. **HIGH** · web `apps/web/components/habits/edit-habit-modal.tsx:183` + mobile `apps/mobile/components/habits/edit-habit-modal.tsx:199` · dim 1+5 · Same gating divergence as finding 1 in the edit modal. Same fix.
3. **HIGH** · web `apps/web/components/ui/create-api-key-modal.tsx:27-59,112-117,364-378` + mobile `apps/mobile/components/ui/create-api-key-modal.tsx:103-108,130-132,293-300` · dim 3 (format) · Expiry: web enforces a strict `YYYY-MM-DDTHH:MM[:SS]` pattern and interprets it as UTC (`Date.UTC`); mobile accepts anything `new Date()` parses (free text input) and interprets it as local time. Mobile accepts inputs web rejects, and the same canonical string produces an `expiresAtUtc` shifted by the user's UTC offset. Fix: add a shared `parseApiKeyExpiry(value): string | null` (one pattern, one UTC-vs-local decision) in `packages/shared` and use it on both; give mobile a structured date-time picker or the same strict pattern.
4. **MED** · web `apps/web/components/habits/create-habit-modal.tsx:271-339` + mobile `apps/mobile/components/habits/create-habit-modal.tsx:238-245` + `apps/mobile/components/habits/create-habit-modal/sub-habit-editor.tsx:50-99` · dim 1 · Non-pro sub-habit flow: web hides the sub-habit editor for non-pro users entirely; mobile renders editable rows and re-routes a submit containing typed sub-habits to `/upgrade` (`hasTypedSubHabits` branch has no web counterpart). ## Verdict

**14 findings — 3 HIGH · 10 MED · 1 LOW.**

Core rule logic is genuinely centralized (habitFormSchema/validateHabitFormInput, validateGoalDraftInput, validateTagForm, isValidEmail, setNameRequestSchema are imported by both apps with identical i18n keys). The parity gaps are concentrated in (a) the habit-form submit-gating contract (web schema-validity vs mobile title-presence — the exact class named in issue #107), (b) the API-key expiry format/semantics, and (c) a long tail of shared-constant lengths re-declared as platform literals.
