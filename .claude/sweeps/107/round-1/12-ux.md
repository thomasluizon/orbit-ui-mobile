## Findings

- **F1 · MED** · `apps/web/components/habits/habit-list.tsx` (+ `habit-list/drill-content.tsx`) + `apps/mobile/components/habit-list.tsx:1632-1644` · Error state (check 3): web never renders `drillError` — `hooks/use-drill-navigation.ts:52` sets it but no web component consumes it, so a failed sub-habit fetch renders the "No sub-habits yet + Add" empty prompt (misleading empty-as-error). Mobile renders the error text in the drill panel. · Fix: render `drill.drillError` in the web drill panel exactly like mobile.

- **F2 · MED** · `apps/web/app/(app)/profile/page.tsx:186-198` + `apps/mobile/app/(tabs)/profile.tsx:540-543` · Error state (check 3): web gates raw `error.message` behind `NODE_ENV === 'development'` and shows `t('errors.loadProfile')` in prod; mobile renders `error.message` unconditionally in production (raw network/backend strings on screen). · Fix: mirror web's dev-only gating on mobile (`t('errors.loadProfile')` in prod).

- **F3 · MED** · `apps/web/app/(app)/advanced/page.tsx:208-218` + `apps/mobile/app/advanced.tsx:168-171` · Error state (check 3): create-API-key failure shows fixed i18n copy on web (`orbitMcp.createKeyError`) but raw `err.message` on mobile (`err instanceof Error ? err.message : t('apiKeys.errors.create')`), bypassing even `getErrorMessage`; the two platforms also use different fallback keys for the same operation. · Fix: use one shared friendly key (or `getFriendlyErrorMessage`) on both.

- **F4 · MED** · `apps/web/app/(app)/advanced/page.tsx:46-50` + `apps/mobile/app/advanced.tsx:235-241` · Error state (check 3): web `fetchApiKeys` swallows HTTP errors (`if (!res.ok) return []`), so a 500 renders the "No API keys yet" empty state — the existing `apiKeysQuery.error` branch (lines 338-342) is unreachable for HTTP failures. Mobile's `apiClient` throws and surfaces `orbitMcp.apiKeysError`. · Fix: throw on `!res.ok` in the web fetcher so the designed error branch fires.

- **F5 · MED** · `apps/web/app/(app)/ai-settings/page.tsx:36-40` + `apps/mobile/app/ai-settings.tsx:381-405` · Error state (check 3): user-facts load failure is masked as the "no facts yet — Ask Astra" empty state on BOTH platforms (web: `fetchUserFacts` returns `[]` on `!res.ok`; mobile: no `factsQuery.error` branch in the loading/empty/list chain). Sibling surface (API keys on mobile) has a designed error message. · Fix: add an error branch with friendly copy (+ retry) on both; web fetcher should throw on `!res.ok`.

- **F6 · MED** · `apps/web/app/(app)/ai-settings/page.tsx:91-92` + `apps/mobile/app/ai-settings.tsx:248-251` · Copy/i18n asymmetry: the fact category pill renders the raw English slug uppercased on web (`category.toUpperCase()` → "PREFERENCE" in pt-BR) while mobile translates it (`t('profile.facts.' + norm)` → "PREFERÊNCIA"); keys exist in both locales. · Fix: web should use the same `profile.facts.*` keys.

- **F7 · MED** · `apps/web/app/(app)/error.tsx` + `apps/web/app/(auth)/error.tsx` + missing mobile counterpart (no `ErrorBoundary` match anywhere under `apps/mobile`) · Error state (check 3): web ships designed route-level error boundaries (glyph, i18n copy, retry pill, raw message dev-only); mobile exports no `ErrorBoundary` from any route, so a render error has no designed fallback. Secondary: web `(chat)` segment also lacks an `error.tsx` that `(app)`/`(auth)` have. · Fix: export an `ErrorBoundary` from `apps/mobile/app/_layout.tsx` mirroring the web error screen; add `apps/web/app/(chat)/error.tsx`.

- **F8 · MED** · `apps/web/app/(app)/calendar-sync/page.tsx:412-439,458-471` + `apps/mobile/app/calendar-sync.tsx:577-587,635-644` · Empty + loading states (checks 1-2): web empty = CalendarDays glyph + copy + go-back chip, web loading = spinner + copy; mobile empty = bare text block (no glyph, no action), mobile loading = bare text (no spinner). Same surface, thinner treatment on mobile. · Fix: add glyph + action to the mobile empty block and a spinner to its loading block.

- **F9 · MED** · `apps/web/app/(app)/advanced/page.tsx:590-658` + `apps/mobile/app/advanced.tsx:452-458` · Content asymmetry (check 5): the widget info dialog on web shows the 3 shared steps plus the 4-item `WIDGET_FEATURES` list (icons + copy); mobile shows only the 3 steps concatenated into a plain ConfirmDialog description — the features list (and its i18n copy `profile.widgetHow.feature1-4`) is unreachable on the platform that actually has the widget. · Fix: render the features list on mobile (shared `WIDGET_FEATURES` already exists).

- **F10 · LOW** · web n/a (BFF adapter — no offline gate) + `apps/mobile/app/(tabs)/profile.tsx:931-936` · Copy quality: mobile delete-account offline state borrows `calendarSync.notConnected` as its title and concatenates two button labels (`sendCode` / `confirmDelete`) as the description — wrong-context copy in a destructive flow. · Fix: dedicated offline title/description keys for the delete-account sheet.

- **F11 · LOW** · `apps/web/app/(app)/achievements/page.tsx:52-92` + `apps/mobile/app/achievements.tsx:79-101` · Locked-state asymmetry (check 1): mobile shows a Lock glyph circle above title/description/CTA; the web locked block is text-only + CTA. · Fix: add the lock glyph on web.

- **F12 · LOW** · `apps/web/app/(app)/profile/page.tsx:134` + `apps/mobile/app/(tabs)/profile.tsx:345` · Error copy asymmetry (check 3): data-export failure shows fixed `t('dataExport.error')` on web but `getErrorMessage(err, t('dataExport.error'))` on mobile — backend/raw strings can surface mobile-only for the same operation. · Fix: align both on the fixed friendly key.

- **F13 · LOW** · `apps/web/app/(app)/advanced/page.tsx:174-178,294-304,344-348` + `apps/mobile/app/advanced.tsx:100-101,316-337` · Action-surface asymmetry (check 5): web disables "Create key" until capabilities load (`canCreateScopedKey`) and renders a capabilities error message; mobile keeps the create chip enabled with no `capabilitiesQuery.error` display — failed capabilities load opens the modal with an empty scope list. · Fix: mirror web's gating + error copy on mobile.

- **F14 · LOW** · `apps/web/app/(app)/advanced/page.tsx:441-482` + `apps/mobile/app/advanced.tsx:460-472` · Confirm-surface asymmetry (check 5): API-key revoke confirms inline (status-bad tinted banner with Cancel/Confirm chips) on web but via the kit ConfirmDialog on mobile. Both confirm — presentation of the same destructive confirmation differs. · Fix: pick one (kit ConfirmDialog) for both.
